### 1.2 项目背景与立项依据

当前企业 IT 运维面临四大核心痛点:

1. **日志分散难查**: 多系统日志分散在各自服务器，故障排查需逐台登录，单次故障定位平均耗时 45-90 分钟
2. **告警风暴淹没**: 监控告警缺乏智能降噪，运维团队日均接收 500+ 条告警，其中 60% 以上为无效噪音，真正关键告警被覆盖
3. **故障响应慢**: 从告警到根因定位依赖个人经验，MTTR 高达 2-4 小时，P0 事件平均恢复时间超过 60 分钟
4. **经验难沉淀**: 故障处理经验停留在个人脑中，人员流动导致团队能力断层，相同类型故障反复出现

现有运维工具零散、无法联动，缺乏统一的智能运维平台。经技术评估和管理层讨论，决定立项建设 OpsNexus 智能全栈运维平台。

### 1.4 项目范围

**范围内 (In Scope):**

- 25 个功能模块的完整设计、开发、测试、部署 (详见蓝图 ON-BP-001)
- 7 个业务域的微服务架构实现 (日志域/告警域/事件域/CMDB 域/通知域/AI 域/分析域)
- 微前端架构: Shell + 9 个子应用
- 企业微信深度集成 (多机器人 + 生命周期播报 + 多通道路由)
- Active Directory / LDAP 域控认证集成 (OU 映射角色 + 自动同步)
- 多 AI 大模型接入与管理 (Claude / 通义千问 / DeepSeek)
- 开发环境 / 测试环境 / 生产环境完整部署
- 全部项目文档交付

**范围外 (Out of Scope):**

- 硬件采购 (由基础设施团队负责)
- 网络架构改造 (适配现有网络)
- 业务系统代码修改 (业务方自行推送日志到平台接收端点)
- AI 大模型训练和微调 (使用商业 API)
- 移动端 App (第一期仅 Web + 企微通知)
- 企微双向 AI 对话排查 (降为后续版本)

### 5.2 风险复审机制

- 频率: 每两周 Sprint 回顾会时复审风险清单
- 新风险: 任何团队成员可随时提出，PM 在 24h 内评估入册
- 已关闭: 触发条件消失或应对措施验证有效后可关闭
- 升级: 风险分 ≥6 且应对措施无效 → 升级到 Sponsor

### 6.2 技术决策记录 (ADR)

所有重大技术决策采用 ADR (Architecture Decision Record) 格式记录:

```
ADR-{序号}: {决策标题}
状态: 提议 / 已采纳 / 已弃用
背景: 为什么需要做这个决策
决策: 我们选择了什么
理由: 为什么选这个，比较了哪些方案
后果: 这个决策带来的正面和负面影响
```

ADR 存放在代码仓库 `/docs/adr/` 目录，每次技术评审会产出对应的 ADR。

| 用途 | 工具 | 说明 |
|------|------|------|
| 代码管理 | GitHub | 9 域 Monorepo + PR 审查 |
| 任务跟踪 | Jira / GitHub Issues | WBS 任务对应 Issue |
| 文档协作 | Confluence / Git Repo | 设计文档+ADR+会议纪要 |
| 即时通讯 | 企业微信群 | 日常沟通+告警通知 |
| 设计稿 | Figma | UI/UX 高保真原型 |

### 6.4 变更控制流程

```
需求/范围变更请求
  → PM 登记变更请求 (编号+描述+理由)
  → PM 评估影响 (工期/成本/风险)
  → PO 确认业务必要性
  → PM+PO 双签审批 (小变更) / Sponsor 审批 (大变更)
  → 更新 WBS + 通知相关团队
  → 变更记录归档

判定标准:
  小变更: 影响 ≤ 3 人天，不影响里程碑
  大变更: 影响 > 3 人天 或 影响里程碑日期
```

## 七、发布管理

### 7.1 版本号规范

采用语义化版本 (SemVer): `主版本.次版本.修订号`

```
v1.0.0  — Phase 1 首次生产发布 (W12)
v1.1.0  — Phase 2 增量发布 (W24, 新增事件/驾驶舱/降噪)
v1.2.0  — Phase 3 增量发布 (W36, 新增基线/AI 双轨/CMDB/SLA)
v2.0.0  — Phase 4 全量发布 (W54, 全部模块 + 商业级功能 + 本地大模型)
v2.0.1  — 修订版本 (Bug 修复)
v2.1.0  — 延后功能发布 (v2.0 上线后 8 周内):
           - ITSM 双向集成 (ServiceNow + Jira)
           - Python SDK + Go SDK
           - 使用量分析 (Telemetry) 面板
```

### 7.2 发布节奏

| 类型 | 频率 | 审批 | 回滚窗口 |
|------|------|------|---------|
| Phase 大版本 | 每 Phase 结束 (3 个月) | PM+TL+PO 三签 | 2 小时 |
| 迭代小版本 | 每 2 周 Sprint 结束 (可选) | TL 签字 | 1 小时 |
| Hotfix 紧急修复 | 随时 | TL 口头确认 | 30 分钟 |

### 7.3 发布流程

```
代码冻结 (发布前 2 天)
  → QA 执行回归测试
  → 通过 → TL 合并到 release 分支
  → CI 自动构建 Docker 镜像
  → 部署到 Staging 环境
  → Staging 冒烟测试 (30min)
  → PM+TL 确认 Go/No-Go
  → 金丝雀发布: 10% 流量 (观察 30min)
  → 全量发布: 100% 流量
  → 发布后监控 (2h 观察期)
  → 确认稳定 → 发布完成
  → 异常 → 一键回滚到上一版本
```

### 7.4 Change Freeze 窗口

- 每个 Phase 最后 1 周为代码冻结期，仅允许 Bug 修复
- 生产环境大版本发布后 48h 内为观察期，禁止新功能发布
- 法定节假日前 3 个工作日起冻结，节后第 2 个工作日解除
