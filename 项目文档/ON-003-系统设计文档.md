## 一、架构总览

### 1.1 核心设计原则

| 原则 | 含义 | 落地方式 |
|------|------|---------|
| 改 A 不碎 B | 任何域的升级不影响其他域 | 9 域独立 DB+独立部署+事件总线通信 |
| 事件驱动 | 域间不直接调用 API | 全部通过 Kafka 事件发布/订阅 |
| 契约先行 | 接口先定义后实现 | OpenAPI Spec+Event JSON Schema+Pact 契约测试 |
| 数据自治 | 每个域拥有自己的数据库 | 7 个独立 PostgreSQL 实例 |
| 降级优先 | 任何外部依赖不可用时核心功能正常 | AI 熔断降级/AD 降级/通道降级 |

### 1.3 技术选型

| 层级 | 选型 | 备选 | 决策理由 |
|------|------|------|---------|
| 后端语言 | Go (Kratos 微服务框架) | Java Spring Boot | Go 更轻量，容器启动快，内存占用低 |
| 前端框架 | React 18 + Ant Design Pro | Vue3 + Element Plus | React 生态更大，Ant Design 运维类 UI 成熟 |
| 微前端 | qiankun | wujie / Module Federation | 国内大厂广泛验证，文档齐全 |
| API 网关 | Kong / APISIX | Nginx + 自定义 | 原生支持 JWT 验证/限流/版本路由 |
| 事件总线 | Kafka 3.x (KRaft) | Pulsar | Kafka 更成熟，KRaft 去 ZooKeeper 简化运维 |
| 流处理 | Flink | Logstash | Flink 有状态处理能力强，适合复杂清洗逻辑 |
| 日志存储 | Elasticsearch 8.x | OpenSearch | ES 8.x 向量搜索原生支持，知识库可复用 |
| 分析存储 | ClickHouse | Doris / StarRocks | ClickHouse 聚合性能极强，社区活跃 |
| 业务数据库 | PostgreSQL 16 (每域独立) | MySQL 8.0 | PG JSONB 支持灵活 Schema (CMDB 自定义维度) |
| 缓存 | Redis 7.x (Sentinel) | — | 会话+缓存+分布式锁 |
| 对象存储 | MinIO | S3 / OSS | 私有部署，冷日志归档 |
| 认证 | Keycloak | Casdoor | 支持 OIDC/LDAP/企微 SSO，成熟稳定 |
| 契约测试 | Pact | Spring Cloud Contract | 跨语言支持 (Go+JS) |
| CI/CD | GitHub Actions | Jenkins | 代码仓库和 CI 一体化 |
| 容器编排 | Kubernetes | Docker Compose | 生产级弹性伸缩 |

## 二、域间通信 — 事件总线设计

### 2.1 核心原则

- 域间 **禁止** 直接 HTTP/gRPC 调用
- 全部通过 Kafka 事件发布/订阅
- 每个事件有明确的 JSON Schema 和版本号
- 消费者必须容忍未知字段 (向前兼容)

### 2.2 Kafka Topic 规划

| Topic | 生产者 | 消费者 | 分区数 | 保留期 |

| opsnexus.incident.created | 事件域 | 通知域, AI 域, 分析域 | 3 | 30天 |
| opsnexus.incident.updated | 事件域 | 通知域, 驾驶舱 | 3 | 30天 |

### 2.4 Schema 版本化策略

- 版本号: 语义化 major.minor (v1.0, v1.1, v2.0)
- minor 升级 (v1.0→v1.1): 仅新增可选字段，消费者向前兼容
- major 升级 (v1.x→v2.0): 破坏性变更，新旧 Topic 并行运行过渡期
- Schema 注册: 所有 Schema 存储在 Git 仓库 `/schemas/` 目录，CI 自动校验

## 三、各域详细设计

### 3.1 日志域 (Log Domain)

#### 数据库 Schema (PostgreSQL — log-db)

```sql
-- 日志源配置
CREATE TABLE log_sources (
    source_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(100) NOT NULL,
    source_type     VARCHAR(50) NOT NULL,  -- host/middleware/network/security/transaction/database/k8s
    collection_method VARCHAR(50) NOT NULL, -- syslog_tcp/syslog_udp/http_push/kafka_consume/webhook
    config          JSONB NOT NULL,         -- 接收端点配置 (端口/Topic/路径/认证)
    parse_rule_id   UUID REFERENCES parse_rules(rule_id),
    enabled         BOOLEAN DEFAULT true,
    created_at      TIMESTAMPTZ DEFAULT now(),
    updated_at      TIMESTAMPTZ DEFAULT now()
);

-- 日志解析规则
CREATE TABLE parse_rules (
    rule_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(100) NOT NULL,
    format_type     VARCHAR(50) NOT NULL,  -- json/syslog/cef/clf/grok/regex/lua
    pattern         TEXT,                   -- Grok/正则表达式
    multiline_rule  JSONB,                  -- 多行合并配置
    field_mapping   JSONB,                  -- 字段映射到标准 Schema
    sample_log      TEXT,                   -- 示例日志 (用于测试)
    enabled         BOOLEAN DEFAULT true,
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 脱敏规则
CREATE TABLE masking_rules (
    rule_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(100) NOT NULL,
    pattern         VARCHAR(500) NOT NULL,  -- 正则匹配
    replacement     VARCHAR(200) NOT NULL,  -- 替换模式
    priority        INT DEFAULT 0,
    enabled         BOOLEAN DEFAULT true
);

-- 留存策略
CREATE TABLE retention_policies (
    policy_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(100) NOT NULL,
    source_type     VARCHAR(50),
    log_level       VARCHAR(20),            -- 按级别设置不同策略
    hot_days        INT DEFAULT 7,
    warm_days       INT DEFAULT 30,
    cold_days       INT DEFAULT 365,
    enabled         BOOLEAN DEFAULT true
);
```

### 3.2 告警域 (Alert Domain)

#### 数据库 Schema (PostgreSQL — alert-db)

```sql
-- 告警规则
CREATE TABLE alert_rules (
    rule_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(200) NOT NULL,
    description     TEXT,
    layer           SMALLINT NOT NULL CHECK (layer BETWEEN 0 AND 5),
    rule_type       VARCHAR(50) NOT NULL,  -- threshold/keyword/frequency/baseline/trend/ai/business
    condition       JSONB NOT NULL,         -- 规则条件 (阈值/关键字/表达式)
    targets         JSONB,                  -- 关联资产/资产组
    severity        VARCHAR(10) NOT NULL,   -- P0/P1/P2/P3/P4
    ironclad        BOOLEAN DEFAULT false,  -- 铁律告警标记
    enabled         BOOLEAN DEFAULT true,
    schedule        JSONB,                  -- 定时生效规则
    cooldown_minutes INT DEFAULT 5,
    created_by      UUID,
    created_at      TIMESTAMPTZ DEFAULT now(),
    updated_at      TIMESTAMPTZ DEFAULT now()
);

-- 告警实例

## 五、安全设计

### 5.1 威胁建模 (STRIDE)

| 域 | 威胁类型 | 威胁描述 | 缓解措施 |
|----|---------|---------|---------|
| 日志域 | 信息泄露 | 日志中含敏感数据被未授权访问 | Flink 流处理脱敏+数据权限隔离+导出审计+水印 |
| 日志域 | 篡改 | 日志被恶意修改掩盖攻击痕迹 | ES 写入后不可修改 (append-only)+冷存储校验 |
| 告警域 | 拒绝服务 | 告警风暴导致系统过载 | 四层降噪+Kafka 缓冲+限流 |
| 告警域 | 绕过 | 攻击者触发维护模式抑制安全告警 | 铁律告警不可抑制+维护模式需审批 |
| 事件域 | 权限提升 | 低权限用户修改事件状态 | 事件操作权限按角色控制 |
| CMDB 域 | 篡改 | 恶意修改资产拓扑导致降噪误判 | CMDB 变更审计+变更通知 |
| 通知域 | 信息泄露 | 通知内容含敏感信息 | 通知消息脱敏+通道加密 |
| AI 域 | Prompt 注入 | 恶意日志内容影响 AI 判断 | 输入消毒+上下文隔离+输出校验 |
| AI 域 | 信息泄露 | 敏感数据发送给第三方 AI API | 送入前脱敏+合同约束 |
| RBAC | 权限提升 | 绕过数据权限看到其他业务数据 | API 层强制数据过滤+审计 |
| 全局 | 仿冒 | 伪造 Kafka 事件 | Kafka SASL_SSL 认证+Topic ACL |

### 5.2 数据分类与隐私影响评估 (DPIA)

| 数据类别 | 敏感等级 | 示例 | 脱敏规则 | 留存期 | 访问控制 |
|---------|:-------:|------|---------|:-----:|---------|
| 系统日志 | 低 | CPU/内存/磁盘 | 不脱敏 | 热7天+冷1年 | 按业务板块隔离 |
| 应用日志 | 中 | 业务流水/错误堆栈 | 脱敏手机号/身份证 | 热7天+冷6月 | 按服务隔离 |
| 交易日志 | 高 | 支付金额/订单号/银行卡 | 银行卡脱敏+金额按角色 | 热30天+冷2年 | 仅支付团队+审计 |
| 操作审计日志 | 高 | 操作人/命令/目标设备 | 不脱敏 (审计需要原文) | 热30天+冷2年 | 仅安全审计员+管理员 |
| 用户信息 | 中 | 姓名/部门/手机号 | 手机号按角色脱敏 | 账号存续期 | 管理员可见 |
| AI 调用记录 | 中 | 输入摘要/输出结果 | 输入原文不存储(仅 hash) | 1年 | 管理员可查 |

### 5.3 密钥与证书管理

| 密钥类型 | 存储方式 | 轮转策略 |
|---------|---------|---------|
| AI API Key | Vault / K8s Secret + AES-256 加密 | 90 天提醒轮转 |
| DB 密码 | K8s Secret | 180 天强制轮转 |
| Kafka SASL 密码 | K8s Secret | 180 天 |
| TLS 证书 | cert-manager 自动管理 | Let's Encrypt 自动续期 |
| JWT 签名密钥 | Keycloak 内部管理 | 365 天轮转 |
| 企微 Webhook Secret | Vault / K8s Secret | 90 天 |

function initChart(container) {
  const dpr = window.devicePixelRatio || 1;
  const chart = echarts.init(container, null, {
    devicePixelRatio: dpr,  // 关键: 让 Canvas 按实际物理像素渲染

#### 7.2.1 Docker Compose 双机热备架构

```
┌─────────────── 虚拟 IP (VIP) ──────────────────┐
│              ops.company.com                     │
└───────────────────┬─────────────────────────────┘
                    │ Keepalived VRRP
          ┌─────────┴─────────┐
          ▼                   ▼
   ┌─────────────┐     ┌─────────────┐
   │  节点 A      │     │  节点 B      │
   │  (主 Master) │────→│  (备 Standby)│
   │              │ 心跳 │              │
   │ 全部服务运行  │     │ 全部服务运行  │
   │ PG 主库      │────→│ PG 从库      │
   │ ES 主分片    │────→│ ES 副本分片  │
   │ Kafka Broker │────→│ Kafka Broker │
   │ Redis Master │────→│ Redis Replica│
   └─────────────┘     └─────────────┘
         │                     │
    写入请求全部走主        只处理只读+同步
    主故障→VIP漂移→备自动接管
```

**核心组件:**
- Keepalived: VIP 漂移 (VRRP 协议)，主节点故障 < 5s 检测 → 备节点接管 VIP
- 心跳服务: opsnexus-ha-agent (Go 轻量进程)，每 2s 互相探测，连续 3 次失败判定故障
- PG 流复制: 同步模式 (synchronous_commit=remote_apply)，数据零丢失
- ES 跨节点副本: 每个 Index 1 primary + 1 replica 分布在两台机器
- Kafka 双 Broker: replication.factor=2, min.insync.replicas=1
- Redis: 主从复制 + Sentinel (两节点各一个 Sentinel + 一个仲裁 Sentinel 运行在 VIP 侧)

│                                                   │
│ 数据同步状态:                                     │
│   PG 流复制延迟:    < 1s  ✅                      │
│   ES 副本同步:      100%  ✅                      │
│   Kafka ISR:        2/2   ✅                      │
│   Redis 复制延迟:   < 1s  ✅                      │
│                                                   │

└───────────────────────────────────────────────────┘
```

**故障切换流程:**

```
主节点故障
  → Keepalived 检测 VRRP 超时 (< 5s)
  → VIP 漂移到备节点
  → 备节点 opsnexus-ha-agent 检测到 VIP 到达
  → 自动执行:
    1. PG: pg_promote() 将从库提升为主库
    2. ES: 自动选举新 Primary Shard
    3. Kafka: ISR 自动切换 Leader
    4. Redis: Sentinel 自动 Failover
    5. 业务服务: 重新连接本地数据库
  → 全部服务恢复 (总切换时间 < 60s)
  → 通知管理员 (企微/邮件): "主备切换已完成"
  → 原主节点恢复后自动变为备节点
```

### 7.3 灾难恢复与备份

| 场景 | 单机模式 | 双机热备 | K8s 集群 |
|------|---------|---------|---------|
| 单服务崩溃 | Docker 自动重启 | Docker 自动重启 | K8s 自动重启 Pod |
| 主机宕机 | 服务中断，等恢复 | **VIP 漂移，< 60s 切换** | K8s 自动调度 |
| 数据库损坏 | 从备份恢复 (1-2h) | **从库自动提升 (< 30s)** | Patroni 自动切换 |
| ES 节点丢失 | 无法自愈 | **副本分片自动恢复** | 副本分片自动恢复 |
| 全部数据丢失 | 从备份恢复 (4-8h) | 从备份恢复 (4-8h) | 从备份恢复 (4-8h) |

**备份策略:**
- PostgreSQL: 每日全量 (pg_dump) + WAL 持续归档，保留 30 天
- Elasticsearch: 每日 Snapshot，保留 7 天
- ClickHouse: 每周全量备份，保留 4 周
- 备份存储: 本地磁盘 + MinIO/NFS (双机模式备份到对端节点)
- 恢复演练: 每季度一次
- **双机模式额外保障**: 备份自动双写 (主备各一份)，任何一台机器全毁数据不丢

### 7.4 生产级性能设计

#### 7.4.1 API 限流与保护

| 层级 | 策略 | 配置 |
|------|------|------|
| 网关全局限流 | 令牌桶 | 总 QPS ≤ 5000/s，单 IP ≤ 200/s |
| 域级限流 | 漏桶 | 每域 ≤ 1000/s (可独立调整) |
| 用户级限流 | 滑动窗口 | 单用户 ≤ 100 请求/min |
| 日志推送限流 | 令牌桶 | Syslog: ≤ 50000 条/s，HTTP: ≤ 10000 条/s |
| AI 调用限流 | 固定窗口 | ≤ 10 并发 (保护 AI API 配额) |

限流触发时返回 HTTP 429 + Retry-After header，前端展示友好提示。

#### 7.4.2 缓存策略

| 缓存对象 | 存储 | TTL | 失效策略 |
|---------|------|:---:|---------|
| 用户 Session / JWT | Redis | 30min (滑动) | 登出时主动删除 |
| RBAC 权限树 | Redis + 本地内存 | 5min | 权限变更时广播失效 |
| CMDB 资产列表 (热数据) | Redis | 2min | 资产变更时 key 失效 |
| 告警规则列表 | 本地内存 | 1min | 规则变更时广播失效 |
| 日志检索缓存 | ES Request Cache | 自动 | Index 刷新时自动失效 |
| SLA 计算结果 | ClickHouse + Redis | 5min | 事件状态变更时失效 |
| AI 分析结果 | PG (ai_call_logs) | 永久 | 不缓存，每次新计算 |
| 前端静态资源 | CDN / Nginx | 30天 | 版本发布时 hash 更新 |

**缓存穿透防护:** 不存在的 Key 写入空值 (TTL 30s) + 布隆过滤器预判。
**缓存击穿防护:** 热点 Key 使用 Redis 分布式锁 (setnx) 防止并发回源。
**缓存雪崩防护:** TTL 加随机偏移 (±20%) 避免同时过期。

#### 7.4.3 数据库优化

```
PostgreSQL 性能规范:
  - 连接池: 每域 HikariCP max_pool_size=20 (9 域合计 ≤ 180 连接)
  - 慢 SQL 阈值: > 200ms 记录到 slow_log → 每周复审
  - 索引策略: 所有 WHERE 条件字段必须有索引，复合查询用复合索引
  - 分区: alerts 表按月分区 (triggered_at)，incidents 表按月分区
  - 自动 VACUUM: 配置 autovacuum_vacuum_cost_limit=400
  - 只读查询: 双机模式下只读请求路由到从库 (减轻主库压力)

Elasticsearch 性能规范:
  - 分片策略: 每个日索引 3 分片 (单分片 ≤ 50GB)
  - Refresh interval: 5s (非实时搜索场景)
  - Force merge: 温数据每日凌晨合并为 1 段
  - 查询缓存: Request Cache 默认开启
  - 聚合超时: 所有聚合查询设置 timeout=30s
  - 深度分页: 禁止 from+size > 10000，改用 search_after

ClickHouse 性能规范:
  - 物化视图: SLA 计算、告警统计等高频查询用物化视图预聚合
  - 查询超时: max_execution_time=60s
  - 内存限制: 单查询 max_memory_usage=10GB
```

#### 7.4.4 Flink 流处理性能

```
日志处理管道:
  - 并行度: 根据日志量自动伸缩 (2-16 TaskSlot)
  - 背压策略: 上游 Kafka 积压 > 10000 条时触发采样 (仅处理 ERROR+WARN)
  - Checkpoint: 每 30s 一次，超时 60s
  - 状态后端: RocksDB (磁盘存储，大状态不 OOM)
  - 反压监控: Flink Dashboard 监控 + 背压超 5min 触发告警

日志推送突增应对:
  - L1: Kafka 分区自动缓冲 (7 天保留)
  - L2: Flink 背压+采样降级 (只处理 ERROR 级别)
  - L3: HTTP 推送端点返回 429 让客户端重试
  - 恢复后自动消费 Kafka 积压数据，补齐全量日志
```

## 十、安全加固设计

### 10.1 双因素认证 (MFA)

```
启用流程:
  用户设置 → 安全 → 启用 MFA
  → 平台生成 TOTP Secret (base32)
  → 展示二维码 (otpauth://totp/OpsNexus:{username}?secret={secret})
  → 用户扫码绑定 Authenticator App
  → 输入 6 位验证码确认绑定
  → 生成 8 个一次性恢复码 → 用户保存

登录流程 (MFA 已启用):
  输入用户名+密码 → 验证通过
  → 跳转 MFA 验证页 → 输入 6 位 TOTP 码
  → 验证通过 → 签发 JWT → 登录成功
  → 验证失败 → "验证码错误" + 可选"使用恢复码"

技术实现:
  Keycloak 原生 OTP Policy → 配置 TOTP (SHA1, 6 digits, 30s period)
  管理员在 Keycloak Realm 设置中可强制特定 Role 开启 MFA
```

### 10.4 API Token 管理

```sql
CREATE TABLE api_tokens (
    token_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL,
    name        VARCHAR(200) NOT NULL,          -- 用途备注
    token_hash  VARCHAR(64) NOT NULL,           -- SHA-256 hash (原文不存储)
    token_prefix VARCHAR(8) NOT NULL,           -- 前 8 位用于识别 "opx_a1b2..."
    scopes      JSONB NOT NULL,                 -- ["read:alerts","write:logs","read:assets"]
    expires_at  TIMESTAMPTZ,                    -- NULL=永不过期
    last_used_at TIMESTAMPTZ,
    last_used_ip VARCHAR(45),
    call_count  BIGINT DEFAULT 0,
    revoked     BOOLEAN DEFAULT false,
    created_at  TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_tokens_hash ON api_tokens(token_hash) WHERE revoked = false;

-- API 认证流程:
-- 请求: Authorization: Bearer opx_a1b2c3d4e5f6...
-- 网关: 提取 token → SHA-256 hash → 查询 api_tokens 表
-- 验证: revoked=false AND (expires_at IS NULL OR expires_at > now())
-- 鉴权: 检查 scopes 是否包含当前 API 所需权限
```
