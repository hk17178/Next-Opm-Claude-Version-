### 1.3 Git 分支策略

```
main          ← 生产分支，仅通过 MR 合并，受保护
├── develop   ← 开发主干，每日集成
├── feature/ON-123-alert-baseline  ← 功能分支 (从 develop 拉出)
├── bugfix/ON-456-timezone-fix     ← Bug 修复
├── release/v1.0.0                 ← 发布分支 (从 develop 拉出)
└── hotfix/v1.0.1                  ← 紧急修复 (从 main 拉出)
```

**分支规则:**
- feature/bugfix → develop: 必须 MR + 至少 1 人 Code Review + CI 通过
- develop → release: TL 发起，全量回归测试通过
- release → main: PM+TL 双签，UAT 通过
- hotfix → main: TL 口头确认，事后补 MR

### 1.5 代码质量要求

| 检查项 | 工具 | 阈值 | 时机 |
|--------|------|------|------|
| Go Lint | golangci-lint | 0 error, 0 warning | 每次 MR |
| Go 单元测试覆盖率 | go test -cover | ≥ 70% | 每次 MR |
| 前端 Lint | ESLint + Prettier | 0 error | 每次 MR |
| 前端单测覆盖率 | Vitest | ≥ 60% | 每次 MR |
| **i18n 完整性检查** | **i18next-scanner + 自定义脚本** | **中英文翻译 Key 100% 对齐，无遗漏** | **每次 MR** |
| **硬编码文案检测** | **ESLint no-chinese-character 规则** | **JSX 中 0 条硬编码中文 (必须用 t() 函数)** | **每次 MR** |
| 依赖漏洞 | Trivy / Snyk | 无 High/Critical | 每次 MR |
| SAST 静态分析 | Semgrep | 无 High | 每次 MR |
| Docker 镜像扫描 | Trivy | 无 Critical | 每次构建 |
| API 契约测试 | Pact | 100% 通过 | 每次 MR |

### 1.6 国际化 (i18n) 开发规范

**硬性规则: 前端代码中禁止出现任何硬编码的中文或英文 UI 文案。**

```jsx
// ❌ 禁止: 硬编码文案
<Button>保存</Button>
<span>告警中心</span>
<Table.Column title="级别" />

// ✅ 正确: 使用 t() 函数
import { useTranslation } from 'react-i18next';
const { t } = useTranslation('alert');
<Button>{t('common.save')}</Button>
<span>{t('alert.list.title')}</span>
<Table.Column title={t('alert.list.column.severity')} />
```

**i18n 开发流程:**

```
1. 开发者在代码中用 t('key') 引用翻译 Key
2. 同步在 zh/*.json 中添加中文翻译
3. 运行 i18next-scanner → 自动提取所有 t() 调用的 Key
4. 对比 zh/ 和 en/ 翻译文件 → 标记缺失的英文翻译
5. CI 检查: 如果 en/ 有缺失 Key → MR 标记 warning (不阻塞，但需尽快补齐)
6. 每 Sprint 结束前: 所有新增 Key 的英文翻译必须补齐
```

**后端 API 国际化规则:**

```
- 错误消息: 返回 i18n Key 而非直接文案 → { "error": "alert.rule.not_found" }
- 告警规则描述: 存储 { "zh": "CPU 超过 90%", "en": "CPU exceeds 90%" }
- 事件标题: 由 AI 生成时根据用户语言偏好选择语言
- API 通过 Accept-Language header 判断语言 → 影响 message 字段语言
```

## 二、CI/CD Pipeline

### 2.1 Pipeline 流程

```
代码推送到 feature 分支
  → Stage 1: Lint + Format Check (30s)
  → Stage 2: 单元测试 + 覆盖率 (2min)
  → Stage 3: 契约测试 Pact (1min)
  → Stage 4: SAST + 依赖扫描 (2min)
  → Stage 5: Docker Build + 镜像扫描 (3min)
  → Stage 6: Helm Chart Lint + Package (1min)
  → 全部通过 → 允许合并 MR

合并到 develop 分支
  → 自动部署到 Dev 环境
  → 冒烟测试 (自动)

合并到 release 分支
  → 自动部署到 Staging 环境
  → 集成测试 (自动) + 性能测试 (手动触发)
  → UAT 测试 (人工)

合并到 main 分支
  → 自动构建生产镜像 (tag: v{version})
  → 手动触发: 金丝雀发布 10%
  → 人工确认: 全量发布 100%
```

## 三、部署手册

### 3.0 部署方式选择 (仅两种)

| 部署模式 | 适用场景 | 节点数 | HA | 命令 |
|---------|---------|:------:|:--:|------|
| Docker Compose | 评估/中小规模生产 | 1-2 台 | 单机无 / 双机主备 | `docker compose up` |
| Helm Chart | 大规模生产 (K8s) | 3+ 节点 | 全冗余 | `helm install opsnexus` |

不提供手动 K8s YAML / Kustomize / 其他部署方式。所有配置通过安装向导 UI 或 values.yaml 完成。

#### 3.0.1 宿主机环境准备 (Docker Compose 模式必做)

**支持的 Linux 发行版:**

| 发行版 | 版本 | 最低配置 (单机) | 推荐配置 (双机每台) |
|--------|------|:-------------:|:-----------------:|
| Rocky Linux | 8.x / 9.x | 8C 16G 200GB SSD | 16C 64G 2TB SSD |
| Ubuntu Server | 22.04 / 24.04 LTS | 8C 16G 200GB SSD | 16C 64G 2TB SSD |
| Debian | 12 (Bookworm) | 8C 16G 200GB SSD | 16C 64G 2TB SSD |

**一键环境准备:**

```bash
# 自动检测发行版，安装 Docker，配置内核参数，放行防火墙
curl -fsSL https://get.opsnexus.io/prepare.sh | bash

# 脚本自动完成:
#   Rocky 8/9: dnf install docker-ce + firewall-cmd + SELinux 配置
#   Ubuntu 22/24: apt install docker.io + ufw allow + AppArmor 配置
#   Debian 12: apt install docker.io + iptables/nftables + AppArmor 配置
#
# 通用内核参数 (自动写入 /etc/sysctl.d/99-opsnexus.conf):
#   vm.max_map_count=262144       (ES 硬性要求)
#   net.core.somaxconn=65535      (高并发连接)
#   fs.file-max=1000000           (文件描述符)
#   vm.swappiness=1               (禁用 swap)
#
# 数据目录创建:
#   /opt/opsnexus/data/{pg,es,kafka,redis,clickhouse,minio}
```

#### PostgreSQL (主从流复制, 每域独立)

```yaml
# 关键配置 (单机/双机通用)
max_connections: 200
shared_buffers: 4GB       # 内存的 25%
effective_cache_size: 12GB # 内存的 75%
work_mem: 64MB
# 双机模式: 主从流复制

max_wal_senders: 5

wal_archive: 开启 → 归档到本地 + 对端节点
pg_dump: 每日凌晨 02:00 全量

### 3.4 发布流程 (SOP)

#### 标准发布 (Phase 大版本)

```
SOP-REL-001: 标准发布流程

前置条件:
  □ Phase UAT 测试报告已签字
  □ 安全扫描报告无 High/Critical
  □ 性能测试报告达标
  □ 回滚方案已确认
  □ 值班人已安排

执行步骤:
  1. [PM] 发起发布审批 → PM+TL+PO 三签
  2. [TL] 合并 release → main，触发生产镜像构建
  3. [TL] 确认镜像构建成功，tag 为 v{x.y.z}
  4. [TL] 通知相关人员: "发布开始，预计 30 分钟"
  5. [数据工程] 执行数据库迁移 (Flyway migrate)
     □ 迁移前备份数据库
     □ 迁移执行成功
  6. [TL] 触发金丝雀发布: 10% 流量
  7. [QA] 金丝雀冒烟测试 (15 分钟)
     □ 核心功能正常
     □ 错误率无上升
     □ 延迟无明显增加
  8. [TL] 确认金丝雀正常 → 全量发布 100%
  9. [QA] 全量冒烟测试 (15 分钟)
  10. [TL] 发布后观察期 2 小时
      □ 平台监控无异常
      □ 错误日志无新增异常
  11. [PM] 发布公告: "v{x.y.z} 发布完成"
  12. [PM] 更新版本记录

异常处理:
  - 金丝雀异常: 立即回滚金丝雀 → 分析原因
  - 全量发布后异常: 执行回滚 SOP-REL-002
  - 数据库迁移失败: 执行迁移回滚 → 终止发布
```

#### 回滚流程

```
SOP-REL-002: 回滚流程

触发条件: 发布后发现功能异常/性能下降/错误率上升
决策人: TL (紧急情况可口头确认)

执行步骤:
  1. [TL] 宣布回滚: "v{x.y.z} 回滚到 v{x.y.z-1}"
  2. 回滚操作:
     Docker Compose: docker compose pull && docker compose up -d (使用旧镜像 tag)
     Helm K8s: kubectl rollout undo deployment/{svc} 或 helm rollback
  3. [数据工程] 数据库回滚 (如有 Schema 变更):
     Flyway undo 或 从备份恢复
  4. [QA] 回滚后冒烟测试
  5. [TL] 确认回滚成功
  6. [PM] 通知相关人员
  7. [TL] 分析回滚原因 → 修复 → 重新走发布流程

回滚时间目标: < 15 分钟 (不含 DB 回滚)
```

### 4.1 测试金字塔

```

### 5.2 故障排查 SOP

#### SOP-OPS-001: 日志接收中断

```
症状: 新日志不再出现在 ES 中
排查步骤:
  1. 检查 Kafka Consumer Lag
     → Lag 持续增长: Flink Job 可能挂了
     → Lag 为 0 但无新数据: 外部日志源可能停止推送
  2. 检查 Flink Dashboard: Job 是否 RUNNING
     → FAILED: 查看 Flink TaskManager 日志
     → RUNNING 但无 throughput: 检查 Kafka Topic 是否有新数据
  3. 检查日志接收端点状态
     → Syslog: netstat -tlnp | grep 514 确认端口监听
     → HTTP: curl http://localhost:8080/api/v1/logs/ingest 返回 200
     → 检查日志源设备的推送配置和网络连通性
  4. 检查 ES 集群状态
     → GET _cluster/health → 非 green 则排查 ES
恢复:
  - Flink Job 失败: 从最新 Checkpoint 重启
  - 接收端口异常: 重启日志域服务 (Kafka 缓冲数据不丢)
  - ES 异常: 参考 SOP-OPS-003
```

#### SOP-OPS-002: 告警通知不达

```
症状: 告警触发但企微没收到通知
排查步骤:
  1. 检查告警是否被降噪
     → 查看告警详情: suppressed=true → 检查降噪规则
  2. 检查通知域日志
     → 通知记录 status=failed → 查看 error_message
  3. 检查企微 Webhook
     → curl 测试 Webhook URL
     → 检查企微 API 限流
  4. 检查 Kafka 事件
     → alert.fired 事件是否发布
     → 通知域是否消费到事件
恢复:
  - 降噪误抑制: 调整降噪规则 + 手动补发
  - Webhook 失败: 检查 URL/限流/网络
  - Kafka 消费失败: 重启通知域消费者
```

#### SOP-OPS-003: ES 集群 Red

```
症状: ES _cluster/health status=red
排查步骤:
  1. GET _cluster/health?level=indices → 找到 red 的 index
  2. GET _cat/shards/{index}?v → 找到 UNASSIGNED 的 shard
  3. 检查磁盘空间: GET _cat/nodes?v&h=name,disk.used_percent
     → > 85%: 清理旧索引 / 扩容
  4. 检查节点状态: GET _cat/nodes?v
     → 节点缺失: 检查节点进程/网络
  5. 尝试手动分配 shard:
     POST _cluster/reroute { "commands": [...] }
恢复:
  - 磁盘满: ILM 强制删除最老索引
  - 节点故障: 重启节点，等待 shard 恢复
  - 数据损坏: 从 Snapshot 恢复
```

kafka-consumer-groups.sh --bootstrap-server kafka:9092 --group opsnexus-alert --describe

psql -h pg-primary -c "SELECT * FROM pg_stat_replication;"

curl -X POST http://localhost:8080/api/v1/logs/ingest -H 'Content-Type: application/json' -d '{"message":"test"}'
