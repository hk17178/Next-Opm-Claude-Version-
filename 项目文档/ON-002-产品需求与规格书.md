## 一、文档概述

### 1.1 目的

本文档是 OpsNexus 平台的完整需求基线，合并了产品需求 (做什么、给谁用) 和软件规格 (怎么验收、量化指标)。所有设计和开发工作以本文档为准。需求变更须按项目总纲 (ON-001) 中的变更控制流程执行。

### 1.2 读者

- 产品负责人: 确认需求完整性和优先级
- 开发团队: 理解每个功能的详细规格
- 测试团队: 据此编写测试用例
- 运维代表: 确认功能满足实际运维场景

### 1.3 需求编号规则

```
FR-{模块编号}-{序号}   功能性需求 (Functional Requirement)
NFR-{类别}-{序号}      非功能性需求 (Non-Functional Requirement)
IR-{序号}              接口需求 (Interface Requirement)
CR-{序号}              合规需求 (Compliance Requirement)
```

## 三、功能性需求

### 模块 1: 统一日志采集与管理

#### 用户故事

- US-1.1: 作为运维工程师，我需要在一个界面上检索所有服务器的日志，以便不再逐台登录
- US-1.2: 作为运维工程师，我需要按时间范围+关键字+主机名+服务名组合条件搜索日志，以便快速定位问题
- US-1.3: 作为系统管理员，我需要配置日志保留策略 (热/温/冷)，以便控制存储成本
- US-1.4: 作为安全审计员，我需要日志不可篡改且保留 6 个月以上，以便满足合规审计
- US-1.5: 作为运维工程师，我需要导出特定时间范围的日志为文件，以便离线分析或提交工单

#### 功能需求

| 编号 | 需求 | 优先级 | 验收标准 |
|------|------|:------:|---------|
| FR-01-001 | 系统支持被动接收日志: Syslog (RFC3164/5424 TCP/UDP)、HTTP/HTTPS API 推送、Kafka 消费 (外部 Kafka)、通用 Webhook | P0 | 日志推送到接收端点后 5 秒内可在平台检索到 |
| FR-01-002 | 系统支持接收网络设备日志 (Syslog)、数据库日志 (Audit log推送)、安全设备日志 (CEF/LEEF over Syslog) | P0 | Syslog RFC3164/5424 TCP/UDP 双协议支持 |
| FR-01-003 | 系统支持接收 Kubernetes 日志: 通过 Fluent Bit/Fluentd DaemonSet 将日志推送到平台 HTTP 端点，或直接消费 K8s 集群的 Kafka Topic | P0 | K8s 集群配置日志转发后 5 秒内可检索 |
| FR-01-004 | 系统支持 CI/CD 管道日志和云平台审计日志 (CloudTrail/Activity Log) 接入 | P1 | 提供标准 Webhook/API 接入方式 |
| FR-01-005 | 接收的日志经过内部 Kafka 缓冲，保证不丢失 | P0 | Kafka replication.factor ≥ 2 (双机) 或 3 (集群)，消费延迟 < 5s |
| FR-01-006 | 日志经过 Flink 流处理进行清洗、标准化、字段提取 | P0 | 输出标准 Schema: timestamp/source/level/host/service/message/extra |
| FR-01-007 | 日志中的敏感信息 (手机号/身份证/银行卡) 在流处理阶段不可逆脱敏 | P0 | 脱敏后原文不可恢复，脱敏规则可配置 |
| FR-01-008 | 日志存储分级: 热 (ES, 7 天) → 温 (ES, 30 天) → 冷 (MinIO/S3, 1 年+) | P0 | ILM 策略自动执行，无需人工干预 |
| FR-01-009 | 提供日志全文检索界面，支持 Lucene 查询语法 | P0 | 常规查询 P95 响应时间 < 3 秒 |
| FR-01-010 | 支持日志导出为 CSV/JSON 文件 | P1 | 导出需要额外权限，导出操作记录审计日志 |
| FR-01-011 | 支持日志上下文查看 (某条日志的前后 N 条) | P0 | 上下文查看响应时间 < 2 秒 |
| FR-01-012 | 支持日志字段统计和可视化 (TOP N/趋势图/分布图) | P1 | 支持至少 3 种图表类型 |

### 模块 3: 次世代智能告警引擎

#### 用户故事

- US-3.1: 作为运维工程师，我需要系统在 CPU 超过 90% 时自动告警，以便及时处理
- US-3.2: 作为运维主管，我需要关键业务 (支付/订单) 的核心指标 (成功率/延迟) 出现异常时立即告警，即使没有人设置固定阈值
- US-3.3: 作为系统管理员，我需要定义"铁律告警"——某些致命场景必须告警且不能被任何降噪规则抑制
- US-3.4: 作为运维主管，我需要看到本周的指标相比上周同期是否有异常变化

#### 功能需求

| 编号 | 需求 | 优先级 | 验收标准 |
|------|------|:------:|---------|
| FR-03-001 | Layer 0 铁律告警: 主机宕机/服务不可用/交易中断/数据库复制中断/安全入侵/证书过期——绝不被任何降噪规则抑制 | P0 | 铁律告警 100% 触达，延迟 < 30s |
| FR-03-002 | Layer 1 静态规则: 支持阈值告警 (>, <, =, 持续 N 分钟) | P0 | 规则创建后 1 分钟内生效 |
| FR-03-003 | Layer 1 静态规则: 支持关键字告警 (日志中出现特定关键字) | P0 | 支持正则表达式匹配 |
| FR-03-004 | Layer 1 静态规则: 支持频率告警 (某事件在 N 分钟内出现 M 次) | P0 | 时间窗口精度 ≤ 1 分钟 |
| FR-03-005 | Layer 2 智能基线: 自动学习历史数据建立动态基线包络 | P0 | 学习期 3-30 天可配置 |
| FR-03-006 | Layer 2 智能基线: 支持自定义采样区间 (1min/5min/15min/30min/1h) | P0 | 采样区间可在告警规则中选择 |
| FR-03-007 | Layer 2 智能基线: 偏离基线包络 N% 触发告警，N 可配置 | P0 | 默认偏离 30%，可按规则调整 |
| FR-03-008 | Layer 2 智能基线: 业务高峰时段 (如 19:00-22:00) 基线自动放宽，不误报 | P0 | 高峰时段和放宽比例可配置 |
| FR-03-009 | Layer 3 趋势告警: 支持周同比/月环比/年同比/自定义区间对比 | P0 | 对比偏差阈值可配置 |
| FR-03-010 | Layer 3 趋势告警: AI 自动生成趋势报告并推送 | P1 | 趋势报告包含异常指标摘要 |
| FR-03-011 | Layer 4 AI 异常检测: 多维度联合异常发现 (多个指标同时异常) | P1 | AI 分析延迟 < 60s |
| FR-03-012 | Layer 4 AI 异常检测: 日志模式突变检测 (某类日志突然大量出现) | P1 | 检测灵敏度可配置 |
| FR-03-013 | Layer 5 业务逻辑告警: 支持交易成功率、支付延迟、登录失败率等业务指标告警 | P0 | 业务指标从日志/API/数据库提取 |
| FR-03-014 | 告警规则支持关联 CMDB 资产组 (如: 仅监控"支付核心链路"资产组) | P0 | 资产组变化时规则自动调整 |
| FR-03-015 | 告警规则支持启用/禁用/定时生效 | P0 | 定时规则精度 ≤ 1 分钟 |
| FR-03-016 | 告警去重: 相同告警指纹在恢复前不重复触发 | P0 | 去重窗口可配置 |

### 模块 4: AI 关联分析中心

#### 用户故事

- US-4.1: 作为运维工程师，当我收到告警时，我需要 AI 自动分析可能的根因并给出处置建议，以便快速定位问题
- US-4.2: 作为运维主管，我需要 AI 能区分"人为操作导致的故障"和"系统自身故障"，以便有针对性地改进流程
- US-4.3: 作为运维工程师，我需要 AI 分析时能看到系统日志和人为操作同一条时间线上的关联

#### 功能需求

| 编号 | 需求 | 优先级 | 验收标准 |
|------|------|:------:|---------|
| FR-04-001 | 告警触发后自动收集关联上下文 (系统日志+人为操作日志) | P0 | 上下文回溯时间窗口可配置 (默认 30min，范围 5min-48h) |
| FR-04-002 | 系统线索: 自动收集告警主机日志、同服务其他实例、关联网络/DB 日志、CMDB 拓扑上下游 | P0 | 根据 CMDB 拓扑自动扩展关联范围 (1-3 跳可配) |
| FR-04-003 | 人为线索: 自动收集堡垒机登录记录、网络设备配置变更、DB DDL 操作、主机 sudo/su、CI/CD 部署、云平台操作、变更系统记录 | P0 | 至少覆盖 6 种操作日志来源 |
| FR-04-004 | 时间线对齐: 将系统事件和人为操作按时间戳排列，标注因果关联 | P0 | 因果关联的置信度可量化 (0-100%) |
| FR-04-005 | 上下文压缩层: 海量日志预处理后压缩为结构化摘要再送入大模型 | P0 | 压缩后 Token 量控制在 2000-5000 范围 |
| FR-04-006 | AI 输出结构化报告: 疑似根因+置信度+证据链+影响范围+处置建议+相似案例 | P0 | JSON 格式可解析 |
| FR-04-007 | 根因五分类: 每次分析自动标注 (人为操作/变更引发/系统故障/外部依赖/待定) | P0 | 分类标签存入事件记录 |
| FR-04-008 | AI 分析前自动脱敏 (手机号/身份证/银行卡/API Key 不送入大模型) | P0 | 脱敏规则可配置，100% 过滤 |
| FR-04-009 | AI 分析结果推送到企微通知 | P0 | 推送延迟 < 60s (从告警触发算起) |
| FR-04-010 | AI 输出旁显示"有用/没用"反馈按钮 | P1 | 反馈数据用于 Prompt 优化 |
